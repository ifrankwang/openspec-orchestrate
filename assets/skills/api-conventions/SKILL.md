## 路径约定

| # | 规则 | 说明 |
|---|------|------|
| 1 | 路径统一 `{context-path}/api/v1/` 前缀 + kebab-case | 如 `{context-path}/api/v1/orders/{orderId}/items` |
| 2 | RESTful 风格 | 禁止 POST 包打天下；GET 查询、POST 创建、PUT/PATCH 更新、DELETE 删除 |

## 接口层职责

接口层（Controller / Handler）职责边界：

| 职责 | 说明 |
|------|------|
| 参数校验 | 校验入参合法性（必填、格式、长度、枚举值），校验失败返回 4xx |
| 调用下层 | 校验通过后调用应用层服务编排业务 |
| 响应转换 | 将业务结果转换为响应格式（DTO） |
| **禁止** | 不写业务逻辑、不直接访问持久化层 |

## 统一响应结构

所有接口统一返回结构，含以下三部分：

- `code`：业务错误码（数字），成功时为约定成功值
- `message`：错误描述，支持动态占位参数
- `data`：业务数据泛型

文件下载等特殊场景可例外。

## 异常响应体系

接口层不直接处理异常，由全局异常处理器统一拦截。HTTP 状态码与异常语义映射：

| 异常语义 | HTTP 状态码 | 含义 |
|---------|------------|------|
| 资源不存在 | 404 | 请求的资源未找到 |
| 状态冲突 | 409 | 如重复提交、状态不允许 |
| 文件类型不合法 | 415 | 不支持的媒体类型 |
| 参数校验失败 | 400 | 输入不合法 |
| 服务端业务异常 | 500 | 业务处理错误 |
| 外部服务异常 | 500 | 依赖的外部服务调用失败 |
| 其他未捕获异常 | 500 | 兜底 |

业务错误码作为响应体中独立数字字段，与 HTTP 状态码解耦。HTTP 码供客户端/网关判断大类，业务错误码供前端精确分支处理。

## 泛化设计原则（非强约束）

以下为推荐模式，不强制：

| 原则 | 说明 |
|------|------|
| 输入校验 | 入参 String 字段限制长度和特殊字符（InfoSec 合规） |
| 文件上传 MIME | 上传接口接口层校验 MIME/Content-Type，允许类型由项目配置决定 |
