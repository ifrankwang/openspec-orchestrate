# 修复阶段

分析报告输出后，经用户确认某（些）修复项后，走此流程。

## 主代理职责

主代理**仅负责编排**——拆分修复项、分派子代理、串行执行流水线、维护迭代计数。**禁止主代理直接 edit/write 任何文件**。

## 串行流水线

用户确认的修复项逐个或分批按以下流水线执行。两步**分别交给独立 general 子代理串行执行**，主代理不得合并或并行：

1. **开发+测试子代理（general，独立）**：按确认的修复项修改文件。单次调用可处理多个文件的联动改动。子代理内串行完成改文件后按改动类型分流验证：
   - 改到 `src/tools/orchestrate.ts`、`scripts/*.sh` → 运行 `bun test` + `bun run typecheck`；脚本另做 `bash -n` 语法检查
   - 纯文档改动（`assets/agents/*.md`、`assets/skills/*/SKILL.md`、本 skill 文件）→ 跳过自动化测试，由复核子代理验证
2. **复核子代理（general，独立）**：审查两个维度
   - 变更正确性：改了什么、是否满足修复项、是否引入回归
   - **三者一致性校验**：工具/skill/agent 修改是否同步，术语、参数名、流程描述、职责划分是否一致

复核不通过 → 回环到第 1 步重做；设迭代上限 3 次。超限则停止，向用户报告已修复项与残留问题。

## 收尾阶段

复核通过后触发，串行执行以下门禁：

1. **用户确认门禁**：主代理汇总已实施修复项，请用户确认后方可执行后续收尾。
2. **文档+版本子代理（general，独立）**：执行两项任务
   - 更新受影响文档（AGENTS.md、SKILL.md 等），融入现有结构，使用结论式叙述，禁止变更式表达
   - 修改 `package.json` 版本号。主代理依据以下准则自动判定版本类型并告知子代理目标版本：

| 变更类型 | 版本变动 | 示例 |
|---------|---------|------|
| 改流程/新增能力/改工具行为 | minor（0.Y.z → 0.Y+1.0） | 新增校验逻辑、改状态转移 |
| 纯文档措辞/bug 修正/注释 | patch（0.Y.z → 0.Y.z+1） | 修复 typo、澄清描述 |
3. **commit & push（主代理 bash）**：
   - `git add -A` → `git commit`（遵循仓库风格） → `git push`
   - 如未设 remote 或 push 失败，向用户报告并停止
4. **清缓存（主代理 bash）**：
   - `rm -rf ~/.cache/opencode/packages/github:ifrankwang/openspec-orchestrate`
   - 忽略目录不存在的错误

## 迭代日志格式

每次回环在输出中记录：

```
[第 N 轮] 修复项: <列表>
结果: 开发完成 / 测试通过 / 复核<通过|不通过>
累积轮次: N/N_max
```

## 完成输出

所有修复项完成（或超限终止）后，输出：

```
## 修复完成

已实施: <修复项列表>
版本变更: <旧版本> → <新版本> (<minor|patch>)
Commit: <hash>
缓存清理: <成功|跳过(原因)>
未达上限中止: <残留问题列表>
```
