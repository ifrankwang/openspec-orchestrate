# 修复阶段

分析报告输出后，经用户确认某（些）修复项后，走此流程。

## 主代理职责

主代理**仅负责编排**——拆分修复项、分派子代理、汇总结果、维护迭代计数。**禁止主代理直接 edit/write 任何文件**。

## 串行流水线

用户确认的修复项逐个或分批按以下流水线执行：

1. **开发子代理（general）**：按确认的修复项修改文件。单次调用可处理多个文件的联动改动。
2. **测试子代理（general）**：按改动类型分流验证
   - 改到 `src/tools/orchestrate.ts`、`scripts/*.sh` → 运行 `bun test` + `bun run typecheck`；脚本另做 `bash -n` 语法检查
   - 纯文档改动（`assets/agents/*.md`、`assets/skills/*/SKILL.md`、本 skill 文件）→ 跳过自动化测试，由复核子代理代替
3. **复核子代理（general）**：审查两个维度
   - 变更正确性：改了什么、是否满足修复项、是否引入回归
   - **三者一致性校验**：工具/skill/agent 修改是否同步，术语、参数名、流程描述、职责划分是否一致
4. 复核不通过 → 回环到第 1 步重做；设迭代上限 3 次。超限则停止，向用户报告已修复项与残留问题

## 迭代日志格式

每次回环在输出中记录：

```
[第 N 轮] 修复项: <列表>
结果: 开发完成 / 测试通过 / 复核<通过|不通过>
累积轮次: N/N_max
```

## 完成输出

所有修复项完成后（或超限终止后），输出：

```
## 修复完成

已实施: <修复项列表>
未达上限中止: <残留问题列表>
```
