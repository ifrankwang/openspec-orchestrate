> **项目规范优先**：本 skill 所列约定为推荐标准。若项目已有明确规范且与本 skill 不一致，以项目规范为准。

## 检查流程（概览）

工具 reviewer 必须依次执行以下检查类别。每项不可跳跃——要么执行并报告结果，要么注明跳过理由：

| 序号 | 检查类别 | 含义 |
|------|---------|------|
| 0 | 工具环境检查 | 确保运行环境就绪，不可用则降级 |
| 1 | 编译检查 | 项目能否成功编译 |
| 2 | 代码格式检查 | 是否符合代码风格标准 |
| 3 | 架构约束检查 | 层间依赖、架构规则是否遵守 |
| 4 | 静态代码分析 | 代码质量规则（复杂度、错误模式等）|
| 5 | 单元测试 + 覆盖率 | 测试是否通过、覆盖是否达标 |
| 6 | 深度扫描 | 扩展静态分析（规则数量更多）|
| 7 | 质量工具配置检查 | 质量门规则/配置未被弱化 |
| 8 | 汇总与提交 | 全部结果汇总为统一 issue 结构 |

具体工具的检查命令和输出解析由各技术栈对应的 quality-gate skill 提供。

## 0. 工具环境检查

在执行工具检查前，先确保工具运行环境就绪。环境检查失败时先按自愈性步骤尝试恢复；不可自愈或自愈失败后，提请用户处理或裁定。用户裁定降级跳过时，在报告中注明降级理由，不阻塞其他检查。

本条检查按问题类型分别处理：**基础设施环境问题**（Docker daemon 等服务级依赖）不生成 issue；**工具/插件缺失**生成 issue（dimension=maintainability）；其他不可用情况（网络不通等）按降级处理，报告注明理由。

## 1-6 确定性检查

每项检查的结果处理原则：

- **通过**：正常记录结果
- **不通过**：生成统一结构 issue，阻塞 task 完成

首次为项目运行深度扫描时，存量违规项可能很多。应对策略：

1. 仅对新增/修改文件中的违规项生成 issue（通过 git diff 过滤）
2. 存量违规项报告但不生成 issue（在报告中标注 `[legacy]` 供后续处理）

## 7. 质量工具配置检查

检查本轮变更中是否包含质量工具规则/配置文件的改动。若包含，逐项检查：

- 规则是否被删除或降级
- 是否新增了过宽的 exclude/include 配置
- 质量工具插件配置是否被弱化（跳过执行、降低阻塞等级）

配置无削弱 → 通过；存在削弱 → 工具层 issue。

## 8. 工具输出 → 统一 issue 结构

每个工具的输出必须翻译为统一 issue 结构：

```json
{
  "file": "<相对路径>",
  "line": <行号>,
  "dimension": "style|architecture|performance|security|maintainability",
  "severity": "Critical|High|Medium|Low|Info",
  "description": "<问题描述>",
  "suggestion": "<修改建议>"
}
```

各技术栈的 quality-gate skill 定义各自工具的输出到 dimension 和 severity 映射关系。

## 9. 汇总与提交

全部检查完成后，提交检查结果（含 issues 列表、passed 状态）。
