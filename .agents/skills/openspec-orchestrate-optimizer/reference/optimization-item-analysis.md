# 模式 B：优化项分析

针对用户直接提出的优化项进行分析与改进建议，不依赖具体 session 导出。

主代理负责编排分析流程、推理与判断，子代理负责重 I/O 读取。

## 触发条件

用户直接提出优化项（针对编排框架、agent 指令、skill 定义、工具行为、流程阶段等的改进或问题诉求），未提供 sessionID。

## Quick start

1. 与用户确认优化项的范围与性质
2. 读取编排规范，提取约束清单用于评估现状
3. 分派子代理并发读取相关文件，汇总当前实现现状
4. 按优化项性质分流（问题导向 / 改进导向），执行对应分析
5. 读取目标文件，提出改进建议
6. 执行输出前自检
7. 输出报告

## 详细流程

### B1. 明确优化项范围

主代理基于用户描述先做初步判定：

- 优化项涉及的组件类型（agent 定义 / skill 定义 / 工具 / 流程阶段 / 配置 / 脚本）
- 具体涉及哪些文件路径（根据组件类型在 `assets/agents/`、`assets/skills/`、`src/tools/`、`.agents/skills/` 等目录中定位）
- 优化项的性质倾向（是否涉及明确的问题现象，还是单纯的改进/增强诉求）

**若描述模糊**（如"编排太慢"、"审查员有问题"），用 question 工具向用户追问：
- 具体涉及哪个阶段或哪个角色？
- 你观察到的具体现象是什么？
- 你期望达到什么状态？

不强行猜测，宁可多问一次。

### B2. 理解编排规范

读取以下两份文档，提取约束清单：

- `openspec-orchestrate` SKILL.md（`assets/skills/openspec-orchestrate/SKILL.md`）
- `openspec-orchestrator` agent 定义（`assets/agents/openspec-orchestrator.md`）

约束清单用于评估"当前实现是否偏离规范""规范本身是否有缺口或不清晰处"。提取范围同模式 A 的 Step 2，但**不需要信号映射**（信号映射是 session 合规分析专用的，用于在 JSON 中查找验证证据），本模式改为"在文件内容中查找实现证据"。

如优化项涉及的组件明显超出编排规范范围（如直接指向某个 reviewer agent 的内部指令），可同时读取该相关文件作为评估基线。

### B3. 收集相关文件并分析现状

根据优化项范围，列出需要读取的文件清单。典型包括：

| 组件类型 | 路径 |
|---------|------|
| agent 定义 | `assets/agents/<name>.md` |
| 编排 skill | `assets/skills/openspec-orchestrate/SKILL.md`（已在 B2 读取） |
| 框架 skill | `assets/skills/<name>/SKILL.md`、`.agents/skills/<name>/SKILL.md` |
| 工具逻辑 | `src/tools/orchestrate.ts`（注意编译后的运行行为可能与源码有出入，需结合 session 中的实际调用结果交叉验证） |
| 配置/脚本 | 如 `scripts/export-session.sh` 等 |

分派 explore 子代理并发读取（遵循 `superpowers:dispatching-parallel-agents`），每个子代理返回所读文件的**结构化现状摘要**：章节划分、关键约束点、与优化项相关的现有实现描述。

主代理汇总子代理返回结果，形成现状画像。

### B4. 判断优化项性质并分流

基于用户的原始描述 + B3 的现状画像，主代理判断优化项性质：

| 性质 | 信号 | 分析方法 |
|------|------|---------|
| 问题导向 | 用户描述了明确的问题现象（卡住、反复出现某错误、违反明确约束、阶段卡死、报错信息） | 5-Why 根因分析（路径 1） |
| 改进导向 | 用户提出增强或优化诉求（提高并行度、增加边界约束、加速某阶段、新增支持能力），现行实现没"错"，只是不够好 | 现状-差距-方案分析（路径 2） |
| 模糊 | 既像问题又像改进，或表述不清 | 用 question 工具向用户确认：你想追溯某个具体问题的根因，还是想做一项改进/增强？得到回答后再选定路径 |

判断时应优先依据用户表述的措辞与意图，而非仅凭现象严重程度。例如"Phase 4 经常卡住"是问题导向（有卡住现象），"我想让 Phase 4 并行度更高"是改进导向（当前并未出错）。

### B5. 执行分析（按性质分流）

#### 路径 1：问题导向 → 5-Why 根因分析

- 从问题表象出发，逐层追问"为什么"
- 每层回答必须基于事实（文件内容、约束清单、配置逻辑、流程描述），不得凭空推断
- 最终落点必须是**系统性根因**——某 skill 中某章节描述不清、某 agent 指令中某步骤缺乏约束力、某工具中某校验逻辑有漏洞、某阶段缺少明确退出条件等。**不得落在个案操作失误**
- 5-Why 推导出的系统性根因若指向某份文档/文件的缺失或错误，必须在报告输出前读取该文件对应章节进行交叉验证——确认该文件是否确实存在描述不清、缺少约束、或逻辑矛盾的问题。根因结论须有文件内容支撑，不得仅凭推理断言文件缺失了什么

#### 路径 2：改进导向 → 现状-差距-方案分析

1. **理解优化动机**：复述并确认用户的优化目标。若目标有多种可能解读，向用户追问
2. **现状分析**：基于 B3 汇总的现状画像，描述与优化项相关的当前实现
3. **目标状态**：基于优化动机，描述期望达成的状态（能力、性能指标、约束、流程行为）
4. **差距识别**：将当前与目标对比，按维度拆解出 gap（缺失能力、薄弱约束、流程冗余、性能瓶颈等）
5. **推荐方案**：针对每个差距，直接给出推荐改动方案。每项需含：
   - 改动范围（具体文件路径 + 章节/行号）
   - 影响评估（受影响的 agent/skill/工具列表，可能波及的下游行为）
   - 风险与权衡（回归风险、稳定性影响、维护成本、与其他约束的兼容性）

不强行 5-Why：若改进诉求没有"问题现象"作为起点，5-Why 会变成"为问而问"，根因落点牵强。本模式只在用户能描述具体问题现象时才走路径 1。

### B6. 改进分析（共用）

产出可落地的改进建议。每条建议必须指名具体文件路径和章节/行号，且须基于目标文件的当前实际内容。改进对象覆盖范围包括但不限于 agents、skills、tools、本 optimizer skill 自身、plugins / prompts / 配置文件、scripts。

### B7. 输出前自检

共用检查项已在主 SKILL.md "输出前自检"中覆盖。本模式额外检查：

- [ ] 路径 1（5-Why）：每条根因是否已读目标文件对应章节交叉验证

### B8. 输出报告

纯段落格式，不落盘。报告格式按优化项性质分流：

#### 问题导向报告模板

```
## 编排优化项分析报告

**优化项**: <用户描述>  **涉及范围**: <文件/组件列表>

根因：<结论>
建议：<文件路径> → <改动方向>
```

#### 改进导向报告模板

```
## 编排优化项分析报告

**优化项**: <用户描述>  **涉及范围**: <文件/组件列表>

方案：<改动方向>
范围：<文件路径+章节/行号>
```

报告输出后，如需对分析结果实施修复，按 `reference/remediation.md` 执行。

## 本模式特有约束

- 不导出 session、不调用 `scripts/export-session.sh`
- 分析阶段不修改任何文件；修复阶段仅由子代理修改，主代理不得直接改代码
- 改进建议必须指名具体文件路径和章节/行号
- 路径 1（5-Why）必须落到系统性根因（skill/agent/工具的设计层面），不落在个案操作失误
- 路径 2 必须给出方案的风险与权衡，并明确推荐
- 性质模糊时必须用 question 工具向用户澄清，不得自行假设

## 本模式 Gotchas

- 改进导向路径不要把 5-Why 强行套到"加一个新能力"上，根因会牵强
- 工具逻辑分析时，源码（`src/tools/orchestrate.ts`）反映的是声明式行为，实际执行行为可能与源码有出入；遇到怀疑时，结合具体调用结果或与用户确认后再下结论
- 优化项常涉及多个组件的联动改动，方案探索时需评估改动对其他 agent/skill 的连锁影响
- 涉及流程阶段改动时，编排状态机（`opx_*` 工具的状态转移逻辑）需作为评估基线，避免建议破坏状态机约束
